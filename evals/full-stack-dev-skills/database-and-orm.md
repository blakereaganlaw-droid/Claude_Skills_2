# Evals — full-stack-dev-skills:database-and-orm

## 1. Positive trigger (should load the skill)
> "Our invoices list endpoint takes 8 seconds in production but is instant locally, and our
> dev database schema doesn't match prod anymore. Fix the data layer."

Expected: skill loads; diagnoses N+1 via the SQL log and fixes with
`selectinload`/`joinedload` or column selects; restores Alembic as the only schema-change
path (dev drift = hand-edited schema); checks indexes from evidence; keeps transactions at
the service boundary.

## 2. Near-miss (should NOT load this skill)
> "Write a SQL query joining these two CSV exports to find unmatched payments."

Expected: analytical SQL over files — `data-tools-skills:duckdb-local-analytics` /
`data-analytics-bi-skills:sql-for-analysts`. If this skill loads, sharpen the app-data-layer
framing.

## 3. Quality rubric
A good response:
- **Does the task:** N+1 identified and fixed at the query, migration discipline restored,
  constraints in the schema, portable-code check for the SQLite→Postgres path.
- **Teaches:** why DB mistakes are permanent (constraints as rules that outlive code),
  migrations as one-way doors, how ORMs hide the SQL that N+1s.
- **Safe:** reads autogenerated migrations before applying, never floats for money, no
  mid-service commits.
