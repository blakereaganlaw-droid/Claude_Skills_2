# Data-file hygiene

## When to use
- Setting up or tidying the files of an analysis project: naming, folders, raw vs processed,
  what gets versioned where.
- Preparing any dataset, workbook, or example to share (email, git, a skill's references) when
  the source contains sensitive data.
- Not for: cleaning the *contents* of a dataset → see `data-analytics-bi-skills:data-cleaning`.
  For git mechanics → see `coding-agent-skills:git-and-code-review`.

## Do it
1. **Name files so they sort and explain themselves.** Pattern:
   `<dataset>_<scope>_<YYYY-MM-DD or YYYYMM>[_v2].<ext>` — e.g.
   `ap-payments_bank-x_2026-06.csv`, `recon_operating-acct_2026-06-30_v2.xlsx`. Rules that pay
   off: ISO dates (they sort), lowercase with hyphens/underscores (no spaces — they break
   scripts and links), the content named for what it *is* not who asked for it, and no `final`/
   `FINAL2` — a date or v-number tells the truth.
2. **Separate raw, processed, and output — one direction of flow:**

```
project/
├── raw/         # exactly as received; never edited (read-only if you can)
├── processed/   # script outputs: cleaned/joined data (parquet/csv)
├── output/      # deliverables: reports, workbooks, charts
├── scripts/     # the code that turns raw → processed → output
└── README.md    # what this is, where raw came from, how to rerun
```

   The invariant: anything in `processed/` or `output/` can be deleted and regenerated by
   `scripts/` from `raw/`. The moment you hand-edit a processed file, that's broken — fix the
   script instead.
3. **Version code in git; version data by immutability.** Scripts, queries, and notebooks → git.
   Data files are usually too big/sensitive for git; version them by *never overwriting*: new
   pull, new dated filename; corrections get `_v2` with a line in the README about why. If a
   deliverable matters, snapshot the exact inputs that produced it.
4. **Classify before anything leaves your machine.** For each file ask: does it contain
   identifiers (account numbers, IBANs, SSNs/tax IDs, customer/supplier names, emails), real
   balances/amounts tied to identifiable parties, or credentials? If yes, it is sensitive —
   sharing it (git, email, tickets, AI tools) needs sanitization first, per your data policy.
5. **Sanitize structurally, not cosmetically.** Replace identifiers with consistent fakes that
   keep the *shape* (account `021000021-4837291` → `NNNNNNNNN-XXXXXXX` or `BANK-A-ACCT-01`),
   shift or bucket amounts if amounts are the sensitive part, keep the columns/structure so the
   example still teaches. Hiding Excel columns, white text, or deleting rows but keeping the
   pivot cache are *not* sanitization — remake the file cleanly from sanitized data.
   `references/hygiene-checklists.md` has a scrub checklist by file type (Excel metadata, CSV,
   PDFs).
6. **Enforce with the repo's guardrails.** In this library: raw or real artifacts live in
   `*.private.md` / `references/*.local.*` (git-ignored by design); committed examples are
   sanitized and structural. Before any commit of data-adjacent files:
   `git diff --staged` and read what's actually going in.
7. **Retire files deliberately.** When a project ends: raw kept per your retention policy,
   one-off intermediates deleted, deliverables archived with their inputs snapshot and README.
   An `archive/` folder beats a graveyard of maybe-important files in the working directory.

## Why / learn
File hygiene is reproducibility plus safety, and both come from the same idea: *files should
carry their own provenance*. A name with dataset, scope, and ISO date answers "what is this and
is it current?" without opening it; the raw/processed/output flow answers "where did this number
come from?" mechanically — the answer is always "rerun the script." That's why hand-editing a
processed file is the cardinal sin: it creates a number whose provenance is a memory. The
sanitization rules come from how leaks actually happen — not hackers, but a real statement
pasted as a "quick example," a hidden Excel tab, a pivot cache retaining source rows after the
rows were deleted. Structural sanitization works because what makes an example useful is its
*shape* (columns, formats, relationships), not the real values; you can always keep the shape
and lose the secrets. And immutable dated files beat clever version schemes because they make
the safest operation — never destroying anything — also the laziest one.

## Common mistakes
- `final_v2_FINAL(3).xlsx` → dated, versioned names; the newest date is the newest file, no archaeology.
- Editing the raw export "just to fix one thing" → raw is evidence; fix in the script, keep raw pristine.
- Hand-patching a processed file before the deadline → the deliverable now disagrees with a rerun; patch the script.
- "Sanitizing" by hiding columns or deleting visible rows in Excel → hidden sheets, pivot caches, and metadata leak; rebuild from sanitized data.
- Committing a real export to git "temporarily" → git history is forever; sanitized examples only, raw in git-ignored paths.
- Emailing the unsanitized workbook because the recipient is internal → policy applies per data class, not per recipient's employer.
- Keeping every intermediate forever → archive deliverables + inputs, delete regenerable intermediates.

## Tailor to your environment
Record your conventions in `references/your-environment.md`: your naming pattern, folder
skeleton, retention rules, what your organization classifies as sensitive, and the approved
sharing channels per class. This library's own guardrails (`.gitignore` patterns
`*.private.md`, `references/*.local.*`) are the model — mirror them in your own projects.
