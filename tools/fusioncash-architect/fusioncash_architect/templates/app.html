Reconciliation ruleset flow visualizer

Oracle shows rulesets as a flat table. Here each ruleset is a top-to-bottom waterfall in execution order, so rule precedence is visible.

Why order matters: rules run in sequence and each matched bank line leaves the pool. A precise 1:1 rule must run before a broad 1:M rule, or the broad rule can consume transactions the exact rule should have matched. Highlighted rows are precedence inversions to review.

    Advanced Criteria builder & SQL simulator

    Write or load an Advanced Criteria expression, provide sample field values, and get a True/False result — locally, without running an AutoRecon job. Comparisons are case-sensitive, matching Oracle.

    Supported: s.FIELD / t.FIELD references, LIKE / NOT LIKE (with % and _ wildcards), =, <>, IN (…), AND, OR, NOT, || concatenation, and parentheses.

    Bank string parsing sandbox

    Paste a raw statement string, apply a parse rule, and see exactly what lands in RECON_REFERENCE. Literal characters in a rule are delimiters/anchors and are not copied to the target — only captured groups are.

    Tokens: (X~) capture to next delimiter/end · (a-b) positions a…b · (N)/(NN) numeric · (A) alpha · (X) any. A leading-zero ladder (0000000000(N), 000000000(NN), …) strips leading zeros; sequence order decides which rule fires first.
